                                                                      amitp Mod Foaly • 6 months ago

                                                                      No, it is not safe to use squared euclidean distance for both the heuristic and the cost. The reason is that the costs are added individually whereas the heuristic is not. Suppose you have two steps, with cost 5 and 7. When you square the costs, you will end up with g() being 5² + 7² = 74. But when you square the heuristic, you will end up with (5 + 7)² = 144. In algebra, (x + y)² ≠ x² + y². This mismatch becomes much worse as the path becomes longer. For a 20-long path with each step cost 5 you have g() = (5²) * 20 = 500, whereas h() = (5 * 20)² = 10000.

                                                                      I would suggest using regular euclidean distance. Square root is not that expensive on modern machines. A* will run much faster by having a more accurate heuristic than by avoiding the square roots.

                                                                              amitp Mod sir_derpalot • 6 months ago

                                                                              If the portals are just to wrap around the edges of the map you can come up with an approximate heuristic by essentially wrapping the heuristic; see this.

                                                                              If the portals are irregularly spaced, things get much harder. You might look at differential heuristics, which use Dijkstra's Algorithm to precalculate “partial” information that can then be used to construct an A* heuristic. The basic idea is that if you want to find a path from A to B, and you have previously used Dijksktra's Algorithm to find the distance A to C, and B to C, then one estimate for distance from (A to B) is (A to C) minus (B to C). You can place these points C in various areas of the map and come up with several distance estimates, then take the best one. You would need to perform this map analysis for each set of conditions (swimming, walking, etc.). I have a few links to papers here but don't have a good explanation of the topic yet.

                                                                              If your edges don't have weights, also consider Breadth First Search, which can be optimized pretty well, even for a 100x100 map. The advantage Breadth First Search has is that the nodes don't have to be sorted, and sorting is the slowest part of A*. You can further optimize by batch processing the queue (split it up and have a separate queue for distance 1, 2, 3, etc.). If you batch process, you never have to remove things from a queue, so the only operation is appending to the end of a vector.

                                                                                                            amitp Mod Jonathan Huo • 8 months ago

                                                                                                            We're not taking the square root of hCost, and squaring gCost doesn't quite work. The hCost is only a square root if you use Euclidean distance; there are other heuristics that don't always involve square roots. But when you do use square roots, you can't square gCost and get the same result, because sqrt(d2)^2 + g^2 is not equal to (sqrt(d2) + g)^2.

                                                                                                            It's generally not worth worrying about the square roots. They're fairly fast these days. A better heuristic will save you a lot more cycles than the square root will cost you.

                                                                                                            There's a separate scaling problem where you can and should make sure g and h are referring to the same scale.
